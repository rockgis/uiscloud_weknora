## 2. 백엔드 (Go API 서버)

### 개요
- **포트**: 8080
- **프레임워크**: Gin Web Framework
- **ORM**: GORM (PostgreSQL)
- **핫리로드**: Air (`~/go/bin/air`)
- **로그**: `/tmp/weknora-backend.log`

### 시작 방법
```bash
# 환경변수 로드 후 Air로 실행
set -a && source .env && set +a
export DB_HOST=localhost DB_PORT=5433 DOCREADER_ADDR=localhost:50051 \
       MINIO_ENDPOINT=localhost:9000 REDIS_ADDR=localhost:6379 \
       NEO4J_URI=bolt://localhost:7687 QDRANT_HOST=localhost \
       OTEL_EXPORTER_OTLP_ENDPOINT=localhost:4317
~/go/bin/air
```

### 대화 처리 파이프라인

문서 기반 질의응답이 처리되는 순서:

```
사용자 질문
    │
    ▼
1. PluginRewrite          쿼리 재작성 (선택, enable_rewrite: true)
    │
    ▼
2. PluginSearchParallel   병렬 검색
    ├── PluginSearch       의미 검색 (pgvector BM25)
    └── PluginExtractEntity + PluginSearchEntity (GraphRAG 활성화 시)
    │
    ▼
3. PluginRerank           재순위 (enable_rerank: true, rerank_top_k: 5)
    │
    ▼
4. PluginFilterTopK       상위 K개 선택 (embedding_top_k: 10)
    │
    ▼
5. PluginIntoChatMessage  컨텍스트 주입
    │
    ▼
6. PluginChatCompletion   LLM 추론
    │
    ▼
7. PluginStreamFilter     SSE 스트리밍 → 클라이언트
```

### ReACT 에이전트

멀티턴 추론이 필요한 복잡한 질문에 사용:

```
에이전트 엔진 (max_rounds: 5)
├── 사용 가능한 도구:
│   ├── WebSearchTool    외부 웹 검색 (DuckDuckGo 기본)
│   ├── WebFetchTool     URL 내용 가져오기
│   ├── ChatTool         지식 베이스 검색
│   └── FileFunctionTool 파일 처리
└── 추론 흐름: Thought → Action → Observation → Answer
```

### 주요 설정값 (`config/config.yaml`)

```yaml
server:
  port: 8080

conversation:
  max_rounds: 5           # 에이전트 최대 라운드
  embedding_top_k: 10     # 벡터 검색 상위 결과 수
  vector_threshold: 0.5   # 벡터 유사도 임계값
  rerank_top_k: 5         # 재순위 후 선택 수
  rerank_threshold: 0.5   # 재순위 점수 임계값
  enable_rewrite: true    # 쿼리 재작성 활성화
  enable_rerank: true     # 재순위 활성화

knowledge_base:
  chunk_size: 512         # 청크 크기 (토큰)
  chunk_overlap: 50       # 청크 겹침 크기
```

## 4. PostgreSQL (ParadeDB)

### 개요

[ParadeDB](https://www.paradedb.com/)는 PostgreSQL 기반으로 BM25 전문 검색과 pgvector 벡터 검색을 동시에 지원합니다. uiscloud_weknora의 **기본 저장소이자 벡터 검색 엔진**입니다.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `paradedb/paradedb:v0.18.9-pg17` |
| 컨테이너 | `uiscloud_weknora-postgres-dev` |
| 호스트포트 | `5433` (로컬 PG와 충돌 방지) |
| 내부포트 | `5432` |
| 사용자 | `postgres` |
| 암호 | `postgres123!@#` |
| 데이터베이스 | `uiscloud_weknora` |

```bash
# 접속
docker exec -it uiscloud_weknora-postgres-dev psql -U postgres -d uiscloud_weknora
# 또는
psql -h localhost -p 5433 -U postgres -d uiscloud_weknora
```

### 주요 테이블

| 테이블 | 설명 |
|--------|------|
| `tenants` | 테넌트 (멀티테넌시), API 키, 스토리지 쿼터 |
| `users` | 사용자 계정 (username, email, password_hash) |
| `auth_tokens` | JWT 인증 토큰 |
| `models` | LLM/Embedding/Rerank 모델 설정 |
| `knowledge_bases` | 지식 베이스 (청킹 설정, 모델 연결) |
| `knowledges` | 원본 문서 (파일명, URL, 처리 상태) |
| `chunks` | 문서 청크 (내용, 벡터 임베딩, 순서) |
| `sessions` | 대화 세션 |
| `messages` | 대화 메시지 (질문/답변, 참고 문서) |
| `faqs` | FAQ 항목 |
| `tags` | 태그 |
| `mcp_services` | MCP 서비스 연동 설정 |

### 검색 기능

```sql
-- 벡터 유사도 검색 (pgvector)
SELECT * FROM chunks
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;

-- BM25 전문 검색 (ParadeDB)
SELECT * FROM chunks
WHERE chunks @@@ paradedb.match('content', '검색어');
```

### 마이그레이션

```bash
make migrate-up      # 마이그레이션 적용
make migrate-down    # 롤백

# 마이그레이션 파일 위치
migrations/versioned/
├── 000000_init.up.sql        # 초기 스키마 (tenants ~ messages)
├── 000001_agent.up.sql       # 에이전트, 인증, MCP 추가
├── 000002_embeddings.up.sql  # 임베딩 테이블 (pgvector)
└── 000003_chunk_flags.up.sql # 청크 플래그 (deleted, hidden)
```

## 5. Redis

### 개요

3가지 용도로 사용됩니다: 비동기 작업 큐, SSE 스트리밍, LLM 컨텍스트 캐시.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `redis:7.0-alpine` |
| 컨테이너 | `uiscloud_weknora-redis-dev` |
| 포트 | `6379` |
| 암호 | `redis123!@#` |
| DB | `0` |
| 영속성 | AOF (`appendonly yes`) |

```bash
# 접속
docker exec -it uiscloud_weknora-redis-dev redis-cli -a 'redis123!@#'

# 큐 상태 확인 (Asynq)
docker exec -it uiscloud_weknora-redis-dev redis-cli -a 'redis123!@#' keys "asynq:*"
```

### 용도별 상세

#### ① Asynq 비동기 작업 큐
문서 처리, 임베딩 생성 등 시간이 오래 걸리는 작업을 백그라운드로 처리:

```
파일 업로드 API → Asynq 작업 등록 → Redis 큐 저장
                                          ↓
                          Asynq Worker → 문서 파싱 (DocReader gRPC)
                                       → 임베딩 생성 (Embedding 모델)
                                       → PostgreSQL 저장
```

#### ② SSE 스트리밍 (`STREAM_MANAGER_TYPE=redis`)
LLM 응답을 실시간으로 클라이언트에 전달:

```
LLM 토큰 생성 → Redis Pub/Sub → 백엔드 → SSE → 브라우저
키 패턴: stream:{session_id}
```

#### ③ LLM 컨텍스트 캐시
멀티턴 대화 맥락 저장 (24시간 TTL):

```
대화 진행 → 이전 메시지 컨텍스트 → Redis 캐시 (TTL: 24h)
```

## 6. DocReader (gRPC 파서)

### 개요

Python으로 작성된 문서 파싱 전용 마이크로서비스입니다. 다양한 파일 형식을 읽어 텍스트 청크로 분할한 뒤 Go 백엔드에 gRPC로 반환합니다.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `wechatopenai/weknora-docreader:latest` |
| 컨테이너 | `uiscloud_weknora-docreader-dev` |
| 포트 | `50051` (gRPC) |
| 헬스체크 | `grpc_health_probe -addr=:50051` |
| 최대 메시지 | 50MB |

### 지원 파일 형식

| 형식 | 파서 | 비고 |
|------|------|------|
| PDF | pypdfium2 + PaddleOCR | 표/이미지 OCR 지원 |
| Word (docx/doc) | python-docx | |
| Excel (xlsx/csv) | openpyxl / pandas | |
| Markdown / Text | 직접 파싱 | |
| 이미지 (jpg/png) | PaddleOCR | OCR 텍스트 추출 |
| 웹 URL | playwright / requests | 웹 크롤링 |

### gRPC 서비스 정의

```protobuf
service DocReader {
  // 파일 바이너리로부터 파싱
  rpc ReadFromFile(ReadFromFileRequest) returns (ReadResponse);

  // URL로부터 파싱
  rpc ReadFromURL(ReadFromURLRequest) returns (ReadResponse);
}

message ReadResponse {
  repeated Chunk chunks;  // 분할된 청크 목록
  string error;           // 에러 메시지 (빈 문자열이면 성공)
}

message Chunk {
  string content;          // 텍스트 내용
  int32 seq;               // 청크 순서
  int32 start;             // 원본 문서 시작 위치
  int32 end;               // 원본 문서 끝 위치
  repeated Image images;   // 연관된 이미지 목록
}
```

### 파싱 흐름

```
파일 업로드 (Go 백엔드)
    │
    ▼
gRPC ReadFromFile 호출
    │
    ▼
파일 형식 감지 (magika)
    │
    ├─ PDF → pypdfium2 → 레이아웃 분석 → PaddleOCR (이미지/표)
    ├─ DOCX → python-docx → 단락/표 추출
    ├─ XLSX → openpyxl → 셀 데이터 추출
    ├─ MD/TXT → 직접 읽기
    └─ URL → playwright 크롤링 → HTML 파싱
    │
    ▼
청크 분할 (chunk_size=512, overlap=50)
    │
    ▼
이미지 VLM 설명 생성 (선택, enable_multimodal: true)
    │
    ▼
ReadResponse 반환 → Go 백엔드 → 임베딩 생성 → PostgreSQL 저장
```

## 7. Qdrant (벡터 DB)

### 개요

고성능 벡터 데이터베이스입니다. `RETRIEVE_DRIVER=qdrant`로 설정하면 PostgreSQL 대신 벡터 검색 엔진으로 사용됩니다.

> **현재 상태**: 실행 중이나 `RETRIEVE_DRIVER=postgres`이므로 벡터 검색에는 미사용.
> 전환하려면 `.env`에서 `RETRIEVE_DRIVER=qdrant`로 변경 후 백엔드 재시작.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `qdrant/qdrant:v1.16.2` |

### PostgreSQL vs Qdrant 선택 기준

| 항목 | PostgreSQL (pgvector) | Qdrant |
|------|----------------------|--------|
| 기본 설정 | ✅ (`RETRIEVE_DRIVER=postgres`) | 별도 설정 필요 |

## 8. Neo4j (지식 그래프)

### 개요

문서에서 추출한 엔티티와 관계를 그래프로 저장합니다. `ENABLE_GRAPH_RAG=true`로 활성화하면 GraphRAG 검색이 가능합니다.

> **현재 상태**: 실행 중이나 `ENABLE_GRAPH_RAG=false`이므로 비활성.
> 활성화하려면 `.env`에서 `ENABLE_GRAPH_RAG=true`, `NEO4J_ENABLE=true`로 변경.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `neo4j:latest` |
| 컨테이너 | `uiscloud_weknora-neo4j-dev` |
| 브라우저 UI | http://localhost:7474 |
| Bolt (드라이버) | bolt://localhost:7687 |
| 사용자 | `neo4j` |
| 암호 | `.env`의 `NEO4J_PASSWORD` |

```bash
# Neo4j 브라우저에서 접속: http://localhost:7474
# 사용자: neo4j / 암호: .env NEO4J_PASSWORD

# 노드 조회 예시
MATCH (n) RETURN n LIMIT 25;
```

### GraphRAG 동작 방식

```
문서 청크
    │
    ▼
LLM 엔티티/관계 추출
    │  예: ("uiscloud_weknora", "is_a", "RAG 프레임워크")
    │      ("Go", "used_in", "uiscloud_weknora")
    ▼
Neo4j 저장 (노드 + 엣지)
    │
    ▼ (검색 시)
쿼리 엔티티 추출 → Neo4j 그래프 탐색 → 연관 청크 검색
```

## 9. MinIO (오브젝트 스토리지)

### 개요

S3 호환 오브젝트 스토리지입니다. `STORAGE_TYPE=minio`로 설정하면 업로드된 파일을 로컬 디스크 대신 MinIO에 저장합니다.

> **현재 상태**: 실행 중이나 `STORAGE_TYPE=local`이므로 파일은 로컬 디스크에 저장됨.
> 전환하려면 `.env`에서 `STORAGE_TYPE=minio`로 변경 후 백엔드 재시작.

### 접속 정보

| 항목 | 값 |
|------|----|
| 이미지 | `minio/minio:latest` |
| 컨테이너 | `uiscloud_weknora-minio-dev` |
| API | http://localhost:9000 |
| 관리 콘솔 | http://localhost:9001 |
| 사용자 | `minioadmin` |
| 암호 | `minioadmin` |
| 버킷 | `weknora` (자동 생성) |

```bash
# 관리 콘솔에서 직접 파일 탐색: http://localhost:9001
# 사용자: minioadmin / 암호: minioadmin
```

### 스토리지 타입 비교

| 항목 | local | MinIO | Tencent COS |
|------|-------|-------|-------------|
| 설정 | `STORAGE_TYPE=local` | `STORAGE_TYPE=minio` | `STORAGE_TYPE=cos` |
| 저장 위치 | 서버 로컬 디스크 | MinIO 컨테이너 | 클라우드 |
| 개발 편의성 | ✅ 최고 | 보통 | 클라우드 자격증명 필요 |
| 확장성 | ❌ 서버 의존 | ✅ 독립적 | ✅ 무제한 |
| 권장 환경 | 개발 | 스테이징 | 프로덕션 |

## 10. Jaeger (분산 트레이싱)

> **현재 상태**: 실행 중. 백엔드 시작 시 `OTEL_EXPORTER_OTLP_ENDPOINT=localhost:4317`이 설정되어 있으면 자동으로 트레이스 전송.

| OTLP gRPC (수신) | localhost:4317 |
| OTLP HTTP (수신) | localhost:4318 |

```bash
# Jaeger UI 접속: http://localhost:16686
```

## 12. 개발 명령어

### 서비스 관리

```bash
# 인프라 시작 (기본: postgres + redis + docreader)
make dev-start

# 전체 서비스 시작 (qdrant, neo4j, minio, jaeger 포함)
docker compose -f docker-compose.dev.yml --profile full up -d

# 서비스 중지
make dev-stop
docker compose -f docker-compose.dev.yml down

# 서비스 상태 확인
docker compose -f docker-compose.dev.yml ps
```

### 백엔드 개발

```bash
make dev-app        # Air 핫리로드 (dev-start 먼저 실행)
make build          # 바이너리 빌드
make test           # 전체 테스트 실행
make lint           # golangci-lint
make fmt            # 코드 포맷팅 (gofumpt)
make docs           # Swagger 문서 재생성
```

### 프론트엔드 개발

```bash
make dev-frontend          # Vite 개발 서버

```

### 데이터베이스

```bash
make migrate-down                       # 롤백
```
